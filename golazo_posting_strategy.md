# 🎯 Golazo Engine — استراتيجية النشر + خطة التنفيذ

> **الحالة:** القرارات مثبّتة — جاهز للبناء
> **النطاق:** مسار الأخبار فقط (RSS → DeepSeek → نشر). مسار المباريات (Pre/Post-Match من api-football) مؤجّل لمرحلة لاحقة.
> **المصادر المدمجة:** محاور الاستراتيجية الـ9 + خطة Claude Code (Part 1/2/3) + قرارات Gehad.

---

## 1. القرارات المثبّتة (Locked Decisions)

| البند | القرار |
|---|---|
| **التوجيه (Routing)** | `5★` → single فوري \| `3–4★` → يتجمّع في الـRoundup \| `<3` → تجاهل |
| **نتائج المباريات** | الروتينية → `تجاهل` \| الكبرى (نهائيات، كلاسيكو، ديربيات، حسم لقب) → تمر كخبر (DeepSeek يحكم) |
| **المنشورات المنفردة** | فقط `5★` تُنشر فوراً كـsingle |
| **تعارض الليل** | لا توجد نافذة هدوء — `5★` يُنشر فوراً (`shareNow`) في أي ساعة |
| **التوزيع** | نفس المحتوى للمنصات الثلاث (Instagram + Facebook + X) |
| **قيد منصة X** | X يسقف الصور بـ4 → الـRoundup على X = الغلاف + أهم 3 بطاقات فقط (4 صور) |
| **توقيت النشر** | إلزامياً وقت الذروة الخوارزمي — الـRoundup الساعة **9:00 م** بتوقيت الرياض |
| **حجم الـRoundup** | غلاف + أهم 5 أخبار = **6 شرائح** (IG/FB) |
| **الروابط** | لا روابط خارجية في المنشور الأساسي — الرابط في أول رد/تعليق (قاعدة خوارزمية معتمدة) |

### الحد الأقصى اليومي (Frequency Caps)
الـ`5★` الفورية **لا تُحتسب** ضمن السقف (قرار Gehad). البنية الناتجة:
- **مخرَج مجدوَل واحد/يوم** = الـRoundup (الساعة 9م).
- **`5★` فورية** = حسب الأحداث، بلا سقف، **لكن** بفاصل أمان أدنى 30 دقيقة بين منشور وآخر (anti-burst).
- **الحجم اليومي الواقعي/منصة** ≈ Roundup + 1–3 أخبار عاجلة = **2–4 منشورات/يوم**. سقف أمان صلب: **6/يوم/منصة**.

---

## 2. مصدر التخزين (State Store)

الـRoundup يحتاج **تجميع** أخبار الـ`3–4★` عبر تشغيلات اليوم (كل ساعة) ثم تركيبها 9م → يتطلب مخزن دائم.

- **الأساسي: PostgreSQL على Railway** (موجود عندك — الخيار الطبيعي). يخزّن: الأخبار المؤهّلة، الـimportance، حالة النشر، وسجل منع التكرار (dedup).
- **اختياري: Google Sheet** كنسخة بشرية للمراجعة/الجدولة اليدوية (Google Drive موصول فعلاً). يُستخدم فقط لو احتجت رؤية بصرية — مو مخزن العمل.

> القرار: Postgres = مصدر الحقيقة. Sheet = مرآة اختيارية لاحقاً.

---

## 3. توليد الهاشتاجات (Dynamic)

DeepSeek يقرأ المقال أصلاً → يضيف حقل `hashtags` في مخرجاته (لا حاجة لأداة منفصلة):
- **ديناميكي من الكيانات:** كاس العالم → `#كاس_العالم` \| كلاسيكو → `#كلاسيكو #برشلونة #ريال_مدريد`.
- **ثابت (يُضاف برمجياً):** `#Golazo` + هاشتاج الدوري.
- **الأعداد/منصة:** X = 1–2 \| Instagram = 8–12 (في أول تعليق، لا الكابشن) \| Facebook = 1–2.

## 4. خطة التنفيذ المرتبة

### ✅ Part 1 — الفلترة الذكية (صغير، مستقل — نبدأ به فوراً)
1. تعديل prompt الـDeepSeek: إخراج JSON يضيف `importance` (1–5) + `hashtags` (مصفوفة).
2. قاعدة رفض النتائج الروتينية → `تجاهل`، مع السماح للكبرى كخبر.
3. Code node: parse يحمل `importance` و`hashtags`.
4. n8n **Filter node** بعد الـparse: أبقِ فقط `importance >= 3`.
5. (اختياري) تشديد الـkeyword pre-filter قبل DeepSeek لتوفير تكلفة الـAPI.

> هذا الجزء **لا يعتمد على شيء** — قابل للبناء الآن بالكامل.

### ⏳ Part 2 — الـCarousel = الـDaily Roundup (كبير)
1. **2a — التجميع:** كل تشغيل، خزّن الأخبار `3–4★` في Postgres (مع dedup).
2. **2b — التركيب (9م):** اسحب أهم 5 → ولّد غلاف + بطاقة لكل خبر عبر `buildSvg()` + `@resvg/resvg-js` → روابط صور مستضافة.
3. **2c — القرار:** `5★` → single فوري (مسار منفصل) \| الباقي → الـRoundup.
4. **2d — النشر:** Buffer `create_post` بمصفوفة `assets` (IG/FB = 6 صور \| X = 4 صور).
5. **2e — الموافقة:** Telegram يرسل **album/media group** بدل صورة واحدة → موافقة → نشر متعدد الأصول.

### 🔌 Part 3 — دمج الاستراتيجية (يُلفّ داخل Part 2)
- **التوقيت:** الـRoundup `customScheduled` الساعة 9م \| الـ`5★` `shareNow`.
- **الـCaps:** تطبيق سقف 6/منصة + فاصل 30 دقيقة.
- **الهاشتاجات + قيد الروابط:** تُطبَّق وقت الـrender/post.
